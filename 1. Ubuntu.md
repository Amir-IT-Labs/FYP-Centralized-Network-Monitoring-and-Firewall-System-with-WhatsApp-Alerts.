1. System Implementation
1.1 Ubuntu Server Setup and Configuration
1.1.1 Introduction to Ubuntu Server
Ubuntu Server is a Linux-based operating system widely used for servers due to its stability, security, flexibility, and strong community support.It is commonly deployed in enterprise environments for hosting
web services, monitoring tools, databases, and network applications.Ubuntu Server is lightweight, does not require a graphical interface, and is highly suitable for virtualized environments.In this project,
Ubuntu Server acts as the central monitoring and alert processing server, hosting the monitoring tools, middleware, and Docker-based services required for real-time network monitoring and WhatsApp alerting.
________________________________________
1.1.2 Reason for Choosing Ubuntu Server
Ubuntu Server was selected for the following reasons:
•	Open-source and free to use
•	Stable and reliable for 24/7 server operations
•	Excellent support for Docker and containerized applications
•	Strong networking support and easy static IP configuration
•	Widely used in real-world enterprise and cloud environments
•	Suitable for firewall-protected, private network deployments
These features make Ubuntu Server an ideal choice for implementing a centralized monitoring and alerting system.
________________________________________
1.1.3 Ubuntu Server Virtual Machine Configuration
The Ubuntu Server was deployed as a virtual machine in a VMware-based environment with the following specifications:
•	Operating System: Ubuntu Server
•	Disk Space: 30 GB
•	Network Type: Host-Only (Private Network)
•	Username: amir
Purpose:
•	Hosting alert middleware
•	Running Docker containers
•	Acting as the monitoring server
The Ubuntu Server operates in a private network, meaning it is not directly exposed to the internet. All outbound traffic is routed securely through the MikroTik CHR firewall.
________________________________________
1.1.4 Network Configuration (Static IP Setup)
To ensure consistent connectivity and reliable monitoring, a static IP address was configured on the Ubuntu Server.
Network Details:
•	IP Address: 192.168.100.10
•	Subnet Mask: /24
•	Default Gateway: 192.168.100.1 (MikroTik CHR)
DNS Servers:
•	8.8.8.8
•	1.1.1.1
The static IP configuration was applied using Netplan, which is the default network configuration utility in modern Ubuntu versions.
________________________________________
1.1.5 Netplan Configuration File
The following Netplan configuration file was created:
File Path:
/etc/netplan/static-hostonly.yaml
Configuration Content:
network:
  version: 2
  renderer: networkd
  ethernets:
    ens33:
      dhcp4: no
      addresses:
        - 192.168.100.10/24
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1
      routes:
        - to: default
          via: 192.168.100.1

1.1.6 Purpose of Netplan Configuration
The Netplan configuration serves the following purposes:
•	Disables DHCP to avoid IP changes
•	Assigns a permanent static IP address
•	Defines the default gateway (MikroTik firewall)
•	Configures reliable DNS servers
•	Ensures all internet-bound traffic passes through the firewall
This setup ensures secure, controlled, and predictable network communication.
________________________________________
1.1.7 Commands Used to Create and Apply Netplan Configuration
The following commands were used:
sudo nano /etc/netplan/static-hostonly.yaml
After saving the file:
sudo netplan apply
To verify IP configuration:
ip a
ip route
1.1.8 Docker-Based Services Overview
Docker is installed on the Ubuntu Server to run application services in isolated containers. This approach improves security, scalability, and manageability.
The following containers are running:
•	Uptime Kuma – Network monitoring tool
•	WAHA – WhatsApp HTTP API service
________________________________________
1.1.9 DNS Configuration Inside WAHA Container
Inside the WAHA container, DNS resolution is configured using the following file:
File Path:
/etc/resolv.conf
Configuration Content:
nameserver 192.168.100.1
1.1.10 Purpose of /etc/resolv.conf
The purpose of this configuration is:
•	Direct DNS queries to the MikroTik firewall
•	Ensure controlled and filtered name resolution
•	Allow outbound WhatsApp API communication via NAT
•	Maintain centralized traffic flow through the firewall
This setup guarantees that all container traffic follows the same security path as the Ubuntu host.
________________________________________
1.1.11 Ubuntu Server Access Restriction Configuration
1.1.12 Objective
The objective of this configuration is to restrict administrative access to the Ubuntu Server so that only the authorized Admin PC (IP address: 192.168.100.5) can access the system via SSH and web services.
All other devices within the LAN or external networks are blocked.
________________________________________
1.1.13 SSH Access Restriction
To restrict SSH access, modifications were made in the SSH daemon configuration file.
Configuration File:
/etc/ssh/sshd_config
Configuration Added (at the end of the file):
AllowUsers amir@192.168.100.5
Explanation:
•	Allows only user amir
•	Only from IP address 192.168.100.5
•	Blocks SSH access from all other IP addresses
After modification, the SSH service was restarted:
sudo systemctl restart ssh

________________________________________
1.1.14 Security Outcome
After applying the above configurations:
•	Only the authorized Admin PC (192.168.100.5) can access Ubuntu Server.
•	A single-layer security mechanism (SSH restriction ) is implemented.
This configuration fulfills the project requirement of restricting administrative access to a single authorized system.


1.1.15 Security Considerations
•	Ubuntu Server is placed in a private host-only network
•	No direct public exposure
•	All external communication is routed through MikroTik CHR
•	Static IP ensures stable firewall rules and NAT configuration
•	Containerized services reduce system-level risk

In this stage, Ubuntu Server was successfully deployed and configured as a secure centralized monitoring server. Static IP addressing, gateway routing, DNS configuration, and Docker-based service isolation were implemented to ensure reliability, security, and real-world applicability.
This server forms the backbone of the monitoring, middleware, and WhatsApp alerting system.
